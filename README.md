# Qwen3 Dataset Preprocessing

Detailed dataset rebuild instructions are in `data/README.md`.

Builds combined SFT datasets from:

- `nvidia/Nemotron-Terminal-Corpus` (`skill_based_mixed`)
- `Nanbeige/ToolMind-Web-QA` (`test`)
- `SWE-Factory/DeepSWE-Agent-Kimi-K2-Trajectories-2.8K` (`default`)

For full builds, NVIDIA rows are loaded from all terminal trajectory splits/configs:
`skill_based_easy/train`, `skill_based_medium/train`, and `skill_based_mixed/train`.

## What this pipeline does

- Normalizes each row into a unified `messages` schema.
- Removes reasoning between `<think>...</think>` for all datasets.
- Concatenates all sources and shuffles deterministically (seeded via `shuffle_key`).
- Keeps data in `messages` format (not tokenized) so training can mask user turns.
- Generates demo examples rendered with the `Qwen/Qwen3-4B-Instruct-2507` chat template.
- In `--mode full`, includes all rows from NVIDIA, SWE, and Nanbeige; NVIDIA is treated as a minimum-share source (at least 50% bytes when feasible with full inclusion).

```bash

  --full-output-dir "/wbl-fast/usrs/ethan/small-agent/data/full-dataset"
```

```bash

```

## Output layout

### Demo

`/wbl-fast/usrs/ethan/small-agent/data/demo-dataset`

- `dataset/` - Hugging Face dataset saved to disk
- `metadata.json` - seed, counts, and schema details
- `examples.jsonl` - rendered chat-template examples
- `source_samples.jsonl` - normalized message samples for auditing

### Full

`/wbl-fast/usrs/ethan/small-agent/data/full-dataset`

- `dataset/`
- `metadata.json` (includes pre/post-balance rows/bytes and NVIDIA byte-share metrics)

## Notes on long rows

- No tokenization or max-length filtering is done in preprocessing.
- Filter long rows during training with your tokenizer/runtime settings.

## Multi-node optional helpers

Use the helper scripts if you want to fan out preprocessing jobs:

```bash
./data/scripts/check_nodes.sh /path/to/krafton-dld-public.pem
./data/scripts/preprocess_remote.sh /path/to/krafton-dld-public.pem
```

These scripts are optional. Local processing remains the default path.

## 2-node B200 ms-swift smoke test (full FT)

This repo now includes a smoke-test path for full-parameter fine-tuning of
`LocoreMind/LocoOperator-4B` with `ms-swift` on 2 `b200` nodes.

### 1) Export demo data to ms-swift JSONL

```bash
uv run python data/scripts/export_demo_for_swift.py \
  --input-dir data/demo-dataset/dataset \
  --output-file data/demo-dataset/train.jsonl
```

### 2) Submit the 2-node Slurm job

```bash
mkdir -p logs outputs
sbatch scripts/train_locooperator_full_b200.sbatch
```

### 3) What the job does

- Uses partition `b200`, 2 nodes, 8 GPUs/node.
- Runs `swift sft` in bf16 with full-parameter settings.
- Attempts `max_length=32768` first.
- Enables sequence packing when matching `ms-swift` packing flags are detected.
- Retries once at `16384` context if the 32k attempt fails and `AUTO_FALLBACK_16K=true`.
- Writes all artifacts under the submission directory:
  - Slurm logs: `./logs/train_<jobid>.out|err`
  - Training outputs/checkpoints: `./outputs/<run_id>/`

### 4) Useful overrides

```bash
sbatch --export=ALL,MAX_STEPS=10,MODEL_ID=LocoreMind/LocoOperator-4B,AUTO_FALLBACK_16K=true \
  scripts/train_locooperator_full_b200.sbatch
```

Default key env vars in the sbatch script:

- `IMAGE` (container image path)
- `DATASET_REL_PATH` (default: `data/demo-dataset/train.jsonl`)
- `MAX_STEPS` (default smoke steps: `20`)
- `DEEPSPEED_CONFIG` (default: `zero2`)
- `AUTO_FALLBACK_16K` (default: `true`)
